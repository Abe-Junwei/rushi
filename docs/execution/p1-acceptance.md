# P1 验收（Rushi 本仓）

对照 Jieyu 计划书 **§8 P1：本地校对闭环**（[`如是我闻-本地版改进计划书-2026-05-11.md`](../../../Jieyu/docs/execution/plans/如是我闻-本地版改进计划书-2026-05-11.md)）。

## 与计划书不一致时的本仓口径

计划书 P1 **验收 3** 原文为：「所有编辑操作可追溯到修改记录。」

**本仓（Rushi）采用放宽口径**（产品/研发共同确认后生效）：

| 计划书字面 | Rushi 放宽后 |
|------------|----------------|
| 每一次 UI 编辑（逐字、每次拖边界等）均写入持久化审计 | **不要求**。会话内仍以 **撤销栈** 支撑误操作回退。 |
| 可追溯 | **以「保存批次」为主**：每次「保存到 SQLite」在 `edit_log` 写入一条 **`save_segments`** 记录（含时间戳、语段条数等摘要，见 `apps/desktop/src-tauri/src/p1.rs`）。 |
| 诊断与取证 | **「导出诊断包」**附带 `edit_log` 最近若干条（见 `p4_diagnostic.rs`）及可选整库快照，用于问题复现与粗粒度变更轨迹；**不承诺**逐键审计等价物。 |

若将来需要 **合规级逐条审计**（每次编辑独立落库），应单独立项（通常超出当前 P1 范围），并扩展 `edit_log` schema 与写入路径。

## 手测建议（与根目录 README 一致）

仍按根目录 README **「P1 手测」** 步骤执行；补充建议：

1. 用 **30–60 分钟量级** 真实中文课音频至少跑通 **一遍**（计划书验收 1），关注波形加载与操作流畅度。  
2. 完成编辑后务必 **保存到 SQLite**，再验证关应用重开（验收 2）。  
3. 导出 **TXT / SRT** 与界面正文对照（验收 4）。  
4. 需要追溯时：导出诊断包，查看 `recent_edit_log.tsv` 与（若附带）`rushi.sqlite3`。

在此放宽口径下，**P1 验收 3 视为由「保存批次 + 诊断包」满足**。
