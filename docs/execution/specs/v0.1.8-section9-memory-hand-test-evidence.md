## 症状更正（2026-06-21 · 手测反馈）

> **原假设（有误）**：转写后侧车 ~3GB → scroll 发涩（research 场景 C）。  
> **实际症状**：**打开 File 进入 Editor，波形 + 语段列表加载完成后即卡**，scroll 不跟手；**与是否正在/已经转写无关**（无转写活动时也复现）。

|  implication | 说明 |
|-------------|------|
| 主因更可能在 **WebView 编辑器加载链** | peaks 进 RAM、语段数组、波形 mount、列表虚拟窗、scroll 热路径 |
| 侧车 unload **不能单独解决**「一进 Editor 就卡」 | 即使 sidecar 仅 ~500MB（defer、未 warmup），仍可能涩 |
| §9 H1/H3 数据仍有效 | 证明 unload API 行为；但 **不是** 用户主痛点的完整解释 |

**建议补测（无转写对照）**：

1. 重启 App 或换 File 后 **不要点转写**，等波形 ready → 采样 `bash scripts/sample-rushi-memory-rss.sh W0-editor-no-transcribe`
2. 记：语段数、音频时长、涩度 1–5
3. 与 H1（转写后 sidecar 3.3GB / app 142MB）对比：**若 app RSS 相近且都卡 → WebView；若仅 H1 卡 → 侧车内存压力叠加**

**侧车 kill 对照（A/B · 约 3 分钟）** — Editor 内、**不要点转写**：

```bash
cd "/Users/junwei/开发/Rushi"
export RUSHI_RELEASE_APP="$PWD/apps/desktop/src-tauri/target/release/bundle/macos/如是我闻.app"
open -a "$RUSHI_RELEASE_APP"
# → 打开 5min/100 段 File，波形 ready 后滚 10s，记涩度 A

sleep 2

# → 再滚 10s，记涩度 B（仍不要转写）

pgrep -fl rushi-asr-sidecar || echo "侧车未自动重启"
```

| A 有侧车 ~520MB | B kill 后 | 判定 |
|-----------------|-----------|------|
| **卡** | **卡**（无改善） | **排除 Plan B 常驻侧车主因** → 查 WebView/波形链 |

## 结论

☐ **Go** · ☑ **WARN** · ☐ **No-Go**

Blocker 失败项：

```
MU-H3: unload 逻辑 OK（loaded=null）。即时 RSS 不变；约 30s~2min 后 3366→2680MB（-22%），仍远高于 600MB 目标。
可能：MPS/PyTorch 异步释放 + macOS 延迟归还，非完全无效但未达 acceptance。
```

**2026-06-21 手测摘要**：

- **W0**：5min / **~100 语段** · 未转写 · 加载完即卡 → **小规模也复现**
- **W0-kill A/B**：侧车 520MB → kill 后 0MB · **scroll 前后均卡** → **主因不在侧车 RSS**
- **W2-dev**：`desktop:dev`（`RUSHI_SKIP_BUNDLED_ASR=1`）同 File 也卡 → **纯 WebView/波形代码回归**，非 release-only
- H1 sidecar 3366MB · H3 unload 逻辑 OK · RSS 仍高（**独立 track**，D8 WARN 可签）
